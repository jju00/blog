# 개념

---

> ***파일 업로드를 허용하는 웹 앱들은 유저가 업로드한 파일을 기반으로 파일 안의 데이터를 파싱해 백엔드에 저장하거나 활용***할 수 있다.

ex) 펌웨어 업로드, 네이버 카페 사진 올리기, 크롬 익스텐션을 유저가 추가하는 경우 등

> [!note]
> 확장자, 파일명, 매직바이트 변경까지가 일반적이지만, **==웹 사이트에서 특정 형식을 요구하는 경우에는 그에 맞춰서 변환 후 업로드==**가 필요함
> 

## 파일 업로드 보안 강화 기술

**1. 파일 확장자 검증:** 화이트리스트와 블랙리스트로 나뉜다. 블랙리스트는 우회 방법이 다양하기 때문에, 보다 안전한 화이트리스트 사용을 많이 사용.

**2. 파일 이름 하드코딩:** 사용자가 설정한 파일 이름 대신 해시화된 이름이나 임의의 문자열을 포함한 이름으로 변경한다. 즉, 하드코딩으로 파일의 이름을 서버에서 지정한 경우 파일 이름 조작을 통한 디렉터리 리스팅 등 불가

**3. 파일 이름 검증:** 파일 이름 변경이 불가능하거나 비즈니스 로직에 맞지 않을 경우, 파일 이름을 검증하고 필터링해야 한다. 2번 처럼 하는 게 불가능할 경우, 이름이라도 잘 검증하기. 특히, 디렉토리 변경을 유발할 수 있는 문자들을 제거해야 한다.

**4. Content-Type 검증:** 정해진 MIME-Type의 파일만 업로드가 가능하도록 설정
![[Pasted image 20250622031558.png]]

**5. 파일 권한 변경:** 파일이 실행되지 않도록 Execute(실행) 권한을 삭제.

**6. 파일 저장 위치 보안:** 파일의 위치나 접근 권한을 외부에 노출하지 않도록 보호.

## 파일 업로드 취약점 진단

파일 업로드 취약점 진단은 ==**소스코드에 접근이 가능하지 않는 이상 직접 시도**==해보는 수 밖에는 없다. 이처럼 검증되지 않고 웹앱이 의도하지 않은 소스(Source)들을 마구 사용하는 방식의 블랙박스 테스팅을 퍼징이라고 함.

**진단 목록**
	
- **파일 확장자 변경**
    
- **파일 Content-Type (Mime-type) 변경**
	
> [!note]
> **MIME-type이랑 Content-type**
>         
>    1. **MIME type이란**
>         
>    >**MIME Type (Media Type):** 인터넷에 전달되는 파일 포맷 및 포맷 컨텐츠를 위한 식별자로, 기본적으로 `type/subtype` 구조이다.
>         
>    - `type` 은 파일의 종류를 의미하며, `subtype` 은 파일의 포맷
>    - `application/json` 의 경우, 바이너리 파일 (application) 중 json 포맷의 파일을 의미하는 식별자로서 기능
>    - 추가 세부정보를 보내기 위해 파라미터를 사용할 수도 있다. ex) `text/plain;charset=us-ascii`
> 		![[Pasted image 20250622140400.png]]
>         ex) multipart 예시 - 파일 업로드하는 패킷을 intercept하는 경우
> 	        ![[Pasted image 20250622140641.png]]
>         
>         test.php라는 파일을 업로드 시 패킷
>         
>         - 위의 Content-type은 이 요청 메세지가 어떤 타입인지를 알려줌. 위에 작성했듯이, `multipart`는 다른 MIME type메세지들을 지닌 복합적인 메세지라는 뜻
>         - 업로드하려는 파일은 요청 본문에서 `application/x-php`라는 MIME type으로 정의됨
	
> [!note]
> 2. **Content-type 이란**
> 
> > **Content-type**: 웹 서버에서 HTTP 통신을 통해 전달받은 요청의 Header 내에서, 해당 요청의 Body에 든 데이터 타입에 대한 정보
> 
> 즉, Request Body에 들어가는 데이터 타입을 HTTP Header가 명시할 수 있는데, 이때 사용하는 필드를 `Content-type` 이라 한다.
> 
> 
> 3. **MIME type과 Content-type의 차이**
> 
> - `Content-type` 과 `MIME type` 의 목적성은 데이터의 타입을 명시하는 것에 있다.
> - 하지만 `Content-type` 의 경우 단순히 웹에 국한되어 사용되지만, `MIME type` 의 경우 인터넷 프로토콜에서 사용되므로 더 *상위의 개념이다.*
> - 한마디로, ***HTTP Request의 Header에 있는 Content-type 필드에는 Request Body에 담긴 데이터의 MIME type을 추가***하는 것이다.
>  
>  ⇒ ==즉, MIME type 자체 (예: `text/html` , `application/json`) 는 값이고, 이를 싣는 헤더 필드가 `Accept`·`Content-Type` 이다.==
>     
> ![[Pasted image 20250622152947.png]]
> - Content-type: 헤더 필드
> - MIME-type: `application/x-www-form-urlencoded`

- **php 코드 실행 확인**
    ==⇒ 우회해서 업로드를 성공한 이후에는 ***LFI 취약점***을 통해서 업로드한 파일의 php코드가 실행되는지 확인해야 한다.== 
    
    > 왜냐하면, LFI는 `include()`와 같은 함수를 통해서, 파라미터 등에 저장한 값, 파일의 내용을 바로 포함시키기 때문
    
    ex) 매직 바이트 등을 변경해서 업로드 시 진짜 png가 아니므로, 업로드한 경로 가서 확인해도 깨진 이미지 밖에 안 뜬다. LFI는 파일의 내용을 바로 포함시키므로, `page=upload/test.png`이런 식으로 LFI하게 하면, 무시하고 바로 안에 작성한 php코드가 실행된다.
